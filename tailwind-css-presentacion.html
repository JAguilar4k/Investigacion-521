Tailwind CSS como API de diseno compilada

Utility-first no elimina arquitectura: la mueve a tokens, variantes, componentes y un pipeline que produce CSS estatico, predecible y pequeno.

  • Paradigma Utility-first

Evolucion arquitectonica

BEM/OOCSS vs CSS-in-JS vs Utility-first

Cada enfoque optimiza una dimension distinta: semantica global, colocacion por componente o generacion por uso real.

Dolores mitigados

El problema real no era escribir CSS: era mantenerlo

Bundle infinito

CSS tradicional tiende a crecer aunque el UI cambie. Las clases antiguas quedan vivas porque borrarlas puede romper pantallas no auditadas.

Tailwind genera utilidades detectadas en archivos fuente.

Especificidad rota

La cascada se vuelve una guerra entre selectores, orden de carga y excepciones. El siguiente cambio empieza con miedo.

Utilities mantienen selectores planos y composables.

Clases huerfanas

Nombres semanticos envejecen mal: la clase sigue diciendo hero-primary aunque ya no sea hero ni primary.

La semantica vive en componentes, no en nombres CSS globales.

Under the hood

Como piensa el motor JIT

Tailwind no ejecuta tu app ni parsea JSX como AST de negocio; detecta candidatos de clase en texto fuente y genera CSS para ellos.

// Fragil: Tailwind no ve clases completas
const klass = `text-${tone}-600`;

// Robusto: las clases existen como strings completos
const tones = {
  success: "text-emerald-700 bg-emerald-100",
  risk: "text-rose-700 bg-rose-100",
};

Produccion

La pregunta no es cuantas clases escribimos, sino cuanto CSS enviamos

Tailwind escanea fuentes, genera utilidades usadas y permite que el resultado final sea cacheable, minificado y pequeno.

Checklist de build

  • Definir fuentes con `@source` para que el escaneo sea explicito.
  • Evitar strings dinamicos como `bg-${color}-500`.
  • Minificar `dist/styles.css` antes de entregar.
  • Revisar clases arbitrarias repetidas y convertirlas en tokens.

Configuracion avanzada

De `tailwind.config.js` a CSS-first

Tailwind v4 acerca la configuracion al lenguaje CSS con `@theme`, `@custom-variant`, `@utility` y `@source`.

Tokens por intencion

Nombres como brand, success, risk y focus explican una decision del sistema, no una pantalla temporal.

@theme {
  --color-brand-700: #155e75;
  --color-risk-700: #be123c;
  --shadow-panel: 0 18px 50px -30px rgb(15 23 42 / .45);
}

Variantes como contrato

La variante dark depende de una clase en el root. JavaScript cambia estado; CSS decide como se ve.

@custom-variant dark (&:where(.dark, .dark *));

Plugins y extensiones

Extender Tailwind sin inventar otro framework

Un plugin debe capturar reglas compartidas del equipo. Si exige demasiada explicacion, probablemente es magia nueva.

addUtilities

Utilidades estaticas cuando el lenguaje visual necesita una primitiva repetible.

matchUtilities

Utilidades dinamicas conectadas a valores del tema.

addComponents

Componentes base con cuidado para no ocultar decisiones importantes.

matchVariant

Variantes parametrizadas para estados o atributos del dominio.

WCAG 2.2 AAA

Tailwind ayuda a expresar estados, pero la semantica empieza en HTML

Botones, formularios, tablas, tabs y dialogos mantienen roles nativos o ARIA correcto. Las utilidades hacen visible cada estado.

Foco visible

El foco se define en `@layer base` para que todo control tenga un anillo de alto contraste.

Estado semantico

Tabs usan `role`, `aria-selected`, roving tabindex y paneles enlazados.

Movimiento responsable

Las animaciones respetan `prefers-reduced-motion` desde la capa base.

Patrones senior

Reglas simples para que Tailwind escale

La herramienta es rapida; la diferencia profesional esta en contratos, tokens y revision consistente.

Hacer

  • Mapear variantes a strings completos.
  • Extraer componentes cuando hay intencion estable.
  • Nombrar tokens por dominio: brand, surface, risk.
  • Medir contraste y foco con teclado.

Evitar

  • Interpolar fragmentos de clases.
  • Convertir valores arbitrarios en norma.
  • Usar el CDN en produccion.
  • Crear plugins que reimplementan un framework completo.

Takeaways

  • Utility-first reduce CSS muerto y especificidad accidental.
  • JIT convierte clases usadas en CSS estatico.
  • v4 acelera builds con motor y configuracion CSS-first.
  • Accesibilidad requiere HTML semantico mas variantes de estado.

Notas del orador

Tesis

Abrir con la idea de API de diseno compilada. Tailwind no es ausencia de arquitectura; es arquitectura expresada en tokens y componentes.

Comparativa

No vender Tailwind como bala de plata. Mostrar que cada enfoque paga costos distintos.

Dolores

Conectar con miedo a borrar CSS, overrides crecientes y clases semanticas que ya no describen nada.

JIT

Enfatizar que las clases deben existir como strings completos. Esto explica muchos bugs de produccion.

Performance

Los numeros vienen de benchmarks oficiales; lo importante para DX es el incremental casi instantaneo.

Bundle

Aclarar que las cifras son estimaciones y dependen de tokens, plugins y patrones de uso.

Configuracion

Marcar el cambio de mentalidad en v4 hacia CSS-first y tokens como contrato.

Plugins

Plugin solo cuando captura una regla compartida. Si requiere explicar mucho, probablemente es demasiado magico.

A11y

Tailwind expresa estados; HTML semantico y contraste siguen siendo responsabilidad del equipo.

Patrones

Convertir la charla en criterios de PR y convenciones de equipo.

Demo

Usar el dashboard para aterrizar estado, responsive, dark mode, accesibilidad y clases arbitrarias.

Cerrar con gobierno: Tailwind escala si los componentes y tokens tienen duenos claros.
